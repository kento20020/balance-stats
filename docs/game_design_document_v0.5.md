## 1. ゲームコンセプト

> **どの敵を選ぶかが未来を変える、確率で削り合うフロア制ゲーム**

- 5フロアを登りきることを目指すダンジョン探索ゲーム
- 「敵の選択順が後の選択肢を左右する」意思決定の連鎖が核心
- 戦闘・探索・フロア突破のすべてが確率ベースで統一されている

### 「詰みは無い」と「死」のレイヤー分離

| レイヤー | 原則 |
|---------|------|
| フロア内の進行 | 行き止まりに入っても、進む手段は常にある（詰みは無い） |
| ラン全体 | HPが0になれば**死亡＝1Fから完全リスタート**（パーマデス） |

### テーマの統一

| 場面 | 確率の現れ方 |
|-----|------------|
| 戦闘 | 攻撃するたびに敵が確率で反応する |
| 回復 | 大回復か小回復かは使うまでわからない（ゴールドで「大」確定購入は可能） |
| 宝箱 | 中身は開けるまで完全に不明。効果はルール変更 |
| ゲート | 上のフロアへの通過コストが確率で変わる（ゴールドで確定通過は可能） |

## 2. ラン構造・勝敗・成長

### 2.1 ラン構造
- 1ラン＝**5フロア**（1F〜5F）。5F踏破で**クリア**
- フロア間で状態（HP・ゴールド・宝箱効果）は引き継がれる

### 2.2 死亡＝パーマデス
- HPが0以下で**死亡 → 1Fから完全やり直し**。ゴールド・宝箱効果・進行はすべて失う
- 死亡時に貯まる慰めの通貨は無い（die-to-progress ではない）

### 2.3 成長モデル：ラン内成長 ＋ win-to-progress

| 時間軸 | 何が起きるか | 失われるか |
|--------|------------|:---:|
| **ラン内成長** | 宝箱（ルール変更）とゴールド（sink）で、そのランの中だけ強くなる | 死亡で失う |
| **恒久強化（win-to-progress）** | **5Fクリアに成功した時のみ**、恒久強化を1つ獲得 | 残る |

- 恒久強化は「死んでは貯まらず、勝ってのみ貯まる」
- **クリア時に残ったゴールド量**が、獲得できる恒久強化の質・選択肢の幅に反映される
- → 含意：5Fに向け敵は強くなるが、プレイヤーは**ラン内成長**で対抗する。「敵スケール × ラン内成長」のバランスが5Fの勝率を決める

## 3. ゲームループ

```
ラン開始（1F・HP満タン・ゴールド0）
  ↓
フロア入場 → 全体マップを俯瞰（種別・強敵・宝箱・回復・ゲートの位置は最初から見える）
  ↓
初期3択から1体を選んで戦闘
  ↓
勝利 → ゴールド獲得 → 繋がるノードが解放
  ↓
解放ノードから次を選んで進む（繰り返し）
  ↓
ゲート到達 → 確率イベント（or ゴールドで確定通過）で上のフロアへ
  ↓（5Fまでループ）
5F踏破 → クリア → 恒久強化を獲得
  ↓
（途中でHP0 → 1Fからやり直し）
```

## 4. フロア構造

### 4.1 ツリーテンプレート（全フロア共通）

```
          [フロア入場]
         /     |     \
      [A]     [B]     [C]      ← 初期公開の3択
      ↙↘    ↙↘    ↙↘
  [D1][D2] [D2][D3] [D3][D4]  ← 解放後に詳細が判明
             ↓     ↓
          [ゲート（イベント）]
                ↓
           [次フロアへ]
```

**設計原則**：多親ノード（複数の行1から解放）＝ゲートへの有効ルート／単親ノード（1体からのみ解放）＝行き止まりになりうる。

### 4.2 ツリー形状とフロア対応

| 形 | 構成 | 特徴 | フロア |
|----|------|------|-------|
| 形B | `[3,3]` | 情報量最小・有効1本・明快 | 1F（チュートリアル） |
| 形A | `[3,4]` | 標準・有効2本・行き止まり2 | 2F〜3F |
| 形C | `[3,5]` | 有効3本・選択肢増 | 4F |
| 形D | `[3,4,3]` | 3層・長期HP管理 | 5F |

### 4.3 アンロック接続表（全形・形Dのrow3はドラフト）

**形B `[3,3]`**：n1=A(行止) / n2=A∨B(ゲート) / n3=B∨C(行止)
**形A `[3,4]`**：D1=A(行止) / D2=A∨B(ゲート) / D3=B∨C(ゲート) / D4=C(行止)
**形C `[3,5]`**：n1=A(行止) / n2=A∨B / n3=A∨C / n4=B∨C（n2〜4ゲート） / n5=C(行止)
**形D `[3,4,3]`**：row2はD1〜D4（形Aと同じ親条件、ただしゲート直結でなくrow3へ）。row3＝E1(D1∨D2,行止) / E2(D2∨D3,ゲート) / E3(D3∨D4,行止)。**※row3はドラフト・要確認**

### 4.4 フロア別敵構成（1F〜3Fは確定、4F〜5Fは抽選）

#### 1F：基礎（形B・固定）
| 役割 | 名前 | 体験タイプ | 強敵 |

| 初期B | 確実な敵 | 削り合い（強） | ✗ |
| 初期C | 確率の敵 | 賭け | ✗ |

#### 2F：確率フロア（形A）

## 7. 戦闘システム

### 基本原則
- **プレイヤーのターンのみ**。攻撃するたびに敵の行動テーブルを1回ロール
- 結果は「与ダメ」と「被ダメ」のトレードオフ

### 1ターンの流れ
```
攻撃を選ぶ
  → evade判定（ずれ系のみ）：成立=こちらのダメージ0／不成立=敵に固定ダメージ
  → ramp更新（レース系のみ：counterダメージ +X）
  → 行動テーブルを1回ロール
  → ログ追記 → 双方HP確認（0以下で終了処理）
```

### 行動タイプ
| タイプ | 内容 | 現在 |
|-------|------|:---:|
| `nothing` | 何も起きない | ✓ |
| `counter` | 固定ダメージ | ✓ |
| `heavy_blow` | 大ダメージ | ✓ |
| `evade` | このターンのプレイヤー攻撃を無効化（**ずれ**） | ✓ |
| `guard` / `heal_self` / `poison` / `buff_self` / `debuff_player` | — | 将来 |

### 敵フラグ
| フラグ | 内容 | 体験 |
|-------|------|------|
| `ramp` | 攻撃のたびに `counter` ダメージ +X | **レース** |

## 8. ゴールド経済

### 入手
- 敵撃破で `loot.gold_range` から獲得。フロアが上がるほど増える

### 使い道（sink）
| sink | 効果 |
|------|------|
| ゲート確定通過 | RNGを回避し無傷で通過 |
| スカウト | ロック中ノードのL3を解放前に先読み |
| 回復「大」確定 | 回復ノードの大/小抽選を「大」に固定 |
| 宝箱リロール／追加開封 | 宝箱の中身を引き直す／追加で開ける |
| （クリア時）恒久強化に変換 | 5Fクリア時、残ゴールドが恒久強化の質に反映 |

## 9. イベントシステム

### 9.1 ゲート
「通れない」結果は無い。変わるのは通過コストのみ。

| 結果 | 内容 |
|------|------|
| 安全通過 | ノーコスト |
| 小傷通過 | 小ダメージ |
| 大傷通過 | 大ダメージ |
| 恩恵通過 | 小回復 |
| 特殊通過 | 状態異常・アイテム（将来） |

- 方向性：安全・小傷が大半、大傷・恩恵はレア
- ゴールドで確定通過（無傷）可能

### 9.2 回復
- 大/小の2種。**判定は使用時**。マップでは ❤ で区別なし
- 有効ルート上に必ず1か所配置
- ゴールドで「大」確定購入可能

### 9.3 宝箱（ルール変更・ラン全体持続）
- 敵ノードに付着。中身は完全サプライズ。ゴールドでリロール／追加開封可能

| 宝箱 | 効果 | 刺さる相手 |
|------|------|----------|
| 反撃封じ | `counter` 無効化 | 削り合い・賭け |
| 加速止め | `ramp` 停止 | レース |
| 見切り | `evade` 率半減 | ずれ |
| 重撃軽減 | `heavy_blow` 半減 | 賭け |
| 金鉱 | 撃破ゴールド2倍 | ゴールド経済 |
| 安全弁 | ゲートから「大傷通過」を除外 | ゲート |

| 体験 | 感じ方 | メカニクス |

| 削り合い | 我慢比べ | 毎回確実な `counter`（低分散） |

属性：`tier_range`／`row`／`experience`／`difficulty`。

**row1**：重戦士・炎術師・影忍者・鉄の蟹・電気魚・風霊・炎の魔人・氷の魔人・毒の魔人

## 13. 乱数アーキテクチャ（v0.4で新規）

> バランス検証（共通乱数＝CRN）を成立させるため、乱数構造を設計段階で固定する。後付けすると全数値の検証がやり直しになる。

### 原則：ドメイン別・文脈キー付き独立ストリーム
- 1本の乱数列を共有しない。用途ごとに独立PRNGを持つ：`layout` / `enemyRoll` / `gate` / `treasure` / `heal`
- 各ストリームは `seed = hash(master_seed, domain, context_key)` で初期化（カウンタ／ハッシュベース乱数）
- `context_key` は「いつ引くか」を一意化（例：敵行動ロール＝`floor:node_id:attack_index`）

### 効果
- 「敵HP 10→12」のような**数値変更が、ゲート結果や行動ロールの出目をずらさない**（別ストリーム・同一キーから引くため）
- → 案Aと案Bが**同じダンジョン・同じ出目**を攻略する＝比較の分散が激減（CRN）
- **限界**：botの方策が分岐を変えると、その先のストリームはずれる。CRNが最大に効くのは「選択を変えない数値調整」。選択が変わる比較（sink削除等）では共有分だけ部分的に効く

## 15. バランス検証フレームワーク（v0.4で新規）

### 15.1 思想
- バランスは感覚でなく**botシミュレーション＋統計**で検証する
- 検証は**「面白さ」そのものは測れない**。測れるのは機械的バランスと「選択が効くか」。残りは人間プレイテスト／公開後テレメトリで補う
- どの指標も**bot方策の下での値**。単一botでなく**方策スペクトル**で回し、「どのbotでも同じ結論」を頑健性の条件とする

### 15.2 フェーズと使う手法
| フェーズ | やること | 手法／ツール |
|---|---|---|
| 0 | 合格条件を数値で確定 | — |
| 1 | 粗い数値で独立大量試行。clear率の桁・死亡フロア分布を把握 | 独立試行＋Wilson CI（`phase12_harness.py`） |
| 2 | **スキル幅・選択感度**でコンセプト成立を検証 | botスペクトル比較（同上） |
| 3 | 目標近傍で小差を詰める／sink ROI／貯め込み検出 | **CRN＋McNemar＋アブレーション**（`balance_analysis.py`） |
| 4 | 退化方策の確認／回帰テスト化 | 探索・RL＋ベイズ最適化＋CI組込 |

### 15.3 合格条件（強クリア率は確定・他は併走/暫定）
| 指標 | 目標 | 状態 |
|---|---|---|
| **強プレイのクリア率** | **25〜40%** | **★確定** |
| ランダムプレイのクリア率 | 1〜5% | 暫定（遊びながら調整） |
| スキル幅（強−random） | 十分大きく、CIが0をまたがない | 併走（必ず横目で確認） |
| 選択感度（初手強制の振れ幅） | 有意な振れ幅がある | 併走（必ず横目で確認） |
| 初手（体験別）クリア率差 | ±5pt 以内 | 暫定 |
| 死亡フロア分布 | 特定フロアに集中しない | 暫定 |
| 単一宝箱modの勝率寄与 | ±8pt 以内 | 暫定 |

> **注意**：「強クリア率25〜40%」だけを最適化目標にすると、**運ゲーだがクリア率は合っている＝選択感度ゼロの退屈な解**に落ちる。クリア率を合わせる作業中も、スキル幅と選択感度（[15.4](#154-コンセプト検証の本丸フェーズ2)）を必ず併走させる。

### 15.4 コンセプト検証の本丸（フェーズ2）
- **スキル幅** ＝ 強botのクリア率 − randomのクリア率。小さい／CIが0をまたぐ ＝ 運ゲー（選択が効いていない）
- **選択感度** ＝ 初手の体験タイプを**強制固定**したときのクリア率の振れ幅。ゼロに近い ＝「どの敵を選んでも同じ」＝コンセプト未成立

### 15.5 因果で測る（フェーズ3）
- **sink ROI**：「使ったランの条件付きクリア率」は**交絡**（危険時ほど使うため）。**アブレーション**（baseline vs sink無効化をCRNペアでMcNemar）で因果寄与を測る。Δ≈0かつ低使用率＝死にメカニクス／Δ突出＝事実上必須（選択消失）
- **貯め込み検出**：hoarder vs spender をCRNペアで対戦。パラメータ全域でhoarder優位なら支配戦略。クリア時変換に**逓減**を入れて再走し、支配が消えるか検証

### 15.6 検証運用の前提
- **本番コア（TS）をheadlessで回し、検証もそのコアで行う**のが理想（参照実装と本番が乖離すると「別物で検証した」問題に戻る）。現状のPython参照実装は数値設計を先に安く回す土台
- 統計：clear率はWilson CI、ペア比較はMcNemar（正確二項検定）＋対応あり比率差CI、稀少clear段階では到達フロア・最終HPの順序差（Wilcoxon／対応ありブートストラップ）も併用

### 15.7 数値調整ワークフロー（データ駆動）

数値は事前に確定させず、**外部データに集約して遊びながら実測調整する**。これを成立させる前提を仕様として固定する。

- **数値はコードに直書きしない**。プレイヤー・敵・behavior weight・`ramp`/`evade`・回復・ゲート確率・sinkコスト・`floor_scale`・`difficulty`閾値を**1つの数値データ（JSON等）に集約**する（参照実装の `phase12_harness.py` の `CONFIG` がこの役割）
- **ホットリロード**：再起動なしで数値を差し替えられること。GitHub Pages配信なら URLパラメータ／localStorage での数値セット上書きと相性が良い
- **数値セットのバージョニング**：各ランログに `param_hash` を残す（検証ハーネスに実装済み）。「どの数値で何%だったか」を後から追える
- **調整ループ**：数値セットを変える → 試行 → まずクリア率を強25〜40%帯に寄せる → そのうえでスキル幅・選択感度を確認 → 退屈解なら構造を疑う、を回す

## 17. 拡張性

- **フロア／敵**：データ独立。`tree_shape`・`enemy_pool`への追記で拡張。`tier_range`で出現制限
- **ゲート**：`gate_result_table`を外部定義。フロア別テーブル可
- **イベント**：`type`で識別、ハンドラ追加で新種別（merchant / hidden_room / optional_enemy / curse / rest）
- **ゴールドsink／恒久強化**：sinkは関数として独立。`meta_unlocks`枠でアンロックツリーへ拡張可
- **数値（チューニング対象）**：全数値は外部データに集約され、コード変更なしで差し替え・実測調整できる（[15.7](#157-数値調整ワークフローデータ駆動)）
